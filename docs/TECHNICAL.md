## 1. What BobBee is

BobBee is a **local, single-process Flask web app** that turns a raw list of
territory accounts into a **ranked, cadenced, day-by-day outreach plan** вҖ” who
an IBM seller should email and call, every weekday of the quarter, and why.

The intended user is an IBM **Select Territory seller** (the persona in the app
is Tim Zhou, covering CA/HI/GU/MP) who carries a book in the low thousands of
accounts and needs to know, each morning, exactly what to do.

It is a **self-contained demo**. Every external system a real seller would touch
вҖ” IBM Sales Cloud / ISC, the IBM install base, ZoomInfo, Salesloft, news and
buying signals вҖ” is **mocked with deterministic fake data** generated on the
machine. The app therefore runs anywhere with **no logins, no VPN, no
internet**. The only genuinely external calls are the two optional AI
integrations (В§11, В§12), both of which **fail soft**: if they can't be reached,
the app degrades to deterministic output and keeps working.

### The screens

| Tab | What it shows |
|-----|---------------|
| **Dashboard** | Today's tasks (emails left / calls right, with progress meters and action buttons), an Activity chart with day/week/month/quarter views and prev/next stepping, a Meetings & opportunities tally, and a Book-of-business snapshot. |
| **Schedule** | The quarter as a calendar вҖ” Year / Quarter / Month / Week / Day. Weekdays only. Clicking a day opens a right-hand panel with that day's emails and calls. |
| **Accounts** | The whole book: a searchable, filterable table (account, industry, location, signals) plus a sidebar of pre-made lists (all accounts, one per cadence, plus Leftovers / No contacts / Future quarters). |
| **Cadences** | Each cadence's definition (its ordered email/call steps), its account roster, per-account progress, and a Finished cadences section. |
| **Email** | Today's outbound emails. Draft-all (AI or deterministic), edit, send. Kebab menu per card; click a name for a contextual detail panel. |
| **Call** | Today's call list with contacts, click-to-call, AI pre-call briefs, and a mark-as-called control. |
| **Profile** | w3-style identity card, a territory-coverage choropleth (CA/HI/GU/MP), Personalization (AI writing/cadence prefs), and Settings (About, Appearance/theme, Access, and how-it-works docs). |
| **Ask BobBee** | A floating chat panel (watsonx Assistant) answering product questions. |

## 2. The one hard rule: deterministic vs. AI

This is the single most important thing to understand about the codebase, and it
is enforced visually and architecturally:

> **Blue = deterministic pipeline data. Purple + a вңҰ sparkle = generated by
> watsonx.ai.**

Almost everything in BobBee is **deterministic Python**: the account generation,
scoring, ranking, cadence assignment, the schedule, the territory map, the
book-of-business stats. Given the same inputs it produces byte-identical output
every time, because all randomness is seeded by stable keys (usually the account
name вҖ” see В§6).

**Three things are model-generated**, each clearly scoped, and each **fail-soft
to a deterministic fallback** built from the same data:

1. The **Play + Sales Angle** on an account's detail view (`advise_accounts`).
2. The **pre-call briefs** on the Call tab (`advise_call_brief`).
3. The **email drafts** on the Email tab (`advise_email`) вҖ” the `/api/email_draft`
   route calls watsonx first and falls back to `_deterministic_email` on any
   failure. The response carries a `source: "watsonx" | "deterministic"` field so
   you can tell which produced a given draft.

Because email drafting is fail-soft, an offline run produces the deterministic
template вҖ” still grounded in the account's real data вҖ” with no visible break.

> **Sparkle placement is precise.** The AI marker goes on things a model
> *generated*: the "Draft all emails" / "Generate all briefings" buttons, the
> Play/Angle block, the brief bullets. It does **not** go on
> descriptive-but-purple text like an email card's cadence-and-step label вҖ” that
> is metadata, not model output, and was deliberately left un-sparkled.

Everywhere a number or a decision is shown, the colour tells you whether a model
touched it. When you extend the app, preserve this: if you add a model call,
mark it purple and give it the sparkle; if you add a computed value, keep it
blue.

**Key properties:**

- **One Flask process.** `run_pipeline.py` (~3,200 lines) is the whole backend.
  It binds **loopback only** (`127.0.0.1:5488`) вҖ” never a LAN interface.
- **The UI is one big HTML document** built in `ui_templates.py` (~4,600 lines):
  a `PAGE_TEMPLATE` string with an embedded `<style>` block and inline
  `<script>`. It behaves like a single-page app вҖ” `showPage(name)` toggles
  `.page` sections; there is no client-side router or build step.
- **State is a handful of in-memory dicts** (`_GMA_STATE`, `_STRATEGIZE_STATE`,
  `_FILL_STATE`, `_BOBBY_STATE`, `_PIPELINE_STATE`) plus files on disk. There is
  no database. Durable results live in the sub-apps' `output/` folders as
  `.xlsx` (tabular handoffs) and `Account_Intelligence/output/*.json` (the
  strategy).
- **Sub-apps are real, separate programs** launched as **subprocesses** with CLI
  args (`_launch` / `_launch_and_wait`), never via interactive prompts. Each
  obeys a **file-handoff contract**: read the previous step's `output/latest.xlsx`,
  write your own. This mirrors how the real pipeline was built and keeps each
  stage independently runnable.

## 4. Process & launch lifecycle

`npm start` вҶ’ `run_pipeline.py:main()`. The sequence, and the non-obvious
decisions in it:

### Port reclaim (stable URL)

The app wants to always live at `http://localhost:5488` so you can bookmark it.
`main()`:

1. Reads the preferred port from `BOBBEE_PORT` / `PORT`, default **5488**.
2. If the port is busy, it inspects the owning process. If that process is
   **another BobBee** (`_reclaim_port` matches on `run_pipeline.py` in the
   command line), it SIGTERMs it and takes the port back вҖ” so a forgotten tab
   never moves your URL.
3. If the port is held by **something else**, it falls back to a free port and
   prints the new URL. It never kills a non-BobBee process.

The resolved port is written back to `os.environ["BOBBEE_PORT"]` so the reloader
child (below) agrees on it.

### Live reload

`app.run(use_reloader=True)`. Werkzeug runs `main()` twice: once in the
supervisor, once in the reloaded child (`WERKZEUG_RUN_MAIN=true`). Only the
supervisor's first pass reclaims the port, opens a browser tab, and seeds the
demo вҖ” guarded by `is_reload_child`. Editing any `.py` (including
`ui_templates.py`, where the UI lives) and refreshing shows the change with no
manual restart.

### Demo seed & persistence

On first launch, `_seed_demo_state()` runs the same two workers the Accounts-tab
buttons call (`_run_get_my_accounts` вҶ’ `_run_strategize`), so the app opens on a
**populated book** instead of an empty state. Controlled by
`BOBBEE_SEED_DEMO` (default on):

- With the seed **on**, the launch-time wipe (`_reset_for_fresh_demo`) is
  **skipped**, so imported/strategized data **survives a restart**. The seed
  only rebuilds what's missing on disk вҖ” first build ~minutes (see В§7 on the
  tiering sample size), every restart after that is instant.
- `BOBBEE_SEED_DEMO=0` restores the original behaviour exactly: clean slate on
  every launch, real first-run empty state.

### вҖңTodayвҖқ

Every date-aware code path routes through **`_today()`** (never
`date.today()` directly). `BOBBEE_DEMO_DATE` pins what the app thinks today is:

- unset вҶ’ the real date;
- `monday` вҶ’ the upcoming Monday;
- `YYYY-MM-DD` вҶ’ an exact date.

Cadences only schedule on weekdays, so on a weekend every "today" view is
legitimately empty; the demo pins to a weekday. The client reads the server's
today from `window.__today` (injected into the page) rather than the browser
clock, so a pinned date stays consistent across the dashboard, calendar, and
email/call lists.

## 5. The front end

There is **no framework and no build**. `ui_templates.py` assembles one HTML
string. The important patterns:

- **`showPage(name)`** вҖ” the router. Toggles `.page.active`, updates the nav,
  and calls the page's loader (`fetchAccountsList`, `fetchSchedule`,
  `fetchTodayEmail`, вҖҰ). It also **closes any open right-hand panel**
  (`closeDayPanel` / `closeSidePanel`), so a contextual panel never bleeds from
  one tab onto the next.
- **Data flow** вҖ” pages fetch JSON from the API and render with template
  literals. State lives in a few module-level `let`s (`_acctData`, `_cal`,
  `_emailItems`, `_callItems`, `_mapData`, вҖҰ).
- **Right-hand panels** вҖ” two kinds, both fixed-position and starting *below* the
  56px header (`top:56px`):
  - the **day panel** (Schedule) вҖ” a day's emails/calls;
  - the **side panel** (Email/Call) вҖ” account or contact detail, opened from a
    card's name or its kebab (вӢ®) menu. Opening either shifts the main content
    left (`body.side-panel-open main { padding-right }`) instead of covering it,
    so you keep scrolling the other cards for context.
- **Batch enrichment** вҖ” today's email and call lists need the detail record for
  every account they touch (20вҖ“40/day). Rather than one `/api/accounts/detail`
  request each (which left the tab blank for seconds), they use
  **`/api/accounts/details`** (В§10), a single batched call. Paired with mtime
  caching on the tiering workbook (`_tiering_rows_by_name`), this took the tabs
  from ~6 s to ~0.03 s server-side.
- **Charts are hand-rolled SVG/CSS**, blue-and-purple only, styled after
  carbon-charts (labeled y-axis, gridlines, a zero rule). No charting library.
- **`window.__today`** and **`AI_SPARK`** are the two globals worth knowing:
  the server's date, and the sparkle SVG string kept in JS so a button that
  rebuilds its own label can restore the AI marker.

## 6. The deterministic data engine (`fake_data.py`)

Every "external system" is simulated here. The defining property is
**determinism seeded by stable keys**, via `_rng(namespace, key)` вҖ” an RNG
seeded by a hash of `(namespace, key)`. Two consequences:

- The **same account name always yields the same everything** вҖ” industry,
  revenue, contacts, signals, city вҖ” across processes and restarts. No values
  drift on reload.
- Anything derivable from a name can be **recomputed anywhere** without carrying
  it through the file handoffs. This is exactly how geography works (В§9).

Public generators (all name- or covid-seeded):

| Function | Produces |
|----------|----------|
| `accounts_for_covids(covids, target)` | The book: exactly `target` unique accounts (default `TERRITORY_ACCOUNT_TARGET = 1911`), dealt round-robin across the seller's coverage IDs. Name space is ~42k combinations, so collisions just retry. |
| `state_for_account` / `city_for_account` / `location_for_account` | Territory + city. `_STATES = ["CA"]*12 + ["HI"]*4 + ["GU"]*2 + ["MP"]` вҖ” weighted to Tim's real coverage, so CA dominates and the Pacific islands are a long tail. |
| `company_rollup` | The ISC company-rollup schema вҖ” **reproduced verbatim** (23 columns, including a header with a trailing space). Do not add columns to it; a test enforces the shape (В§16). |
| `zoominfo_enrichment` | Revenue, employees, contacts + which are decision-makers. |
| `signals_for` | News/buying signals (funding, M&A, leadership, expansion). |
| `contacts_for_accounts` | Per-account contacts with titles and phones. |
| `salesloft_cadence_steps` / `salesloft_people_for_cadence` | Cadence step definitions and rosters. |
| `*_install` (`cloud`, `power`, `storage`, вҖҰ) | IBM install-base footprint per account, used for whitespace. |

> **Design note вҖ” the ISC schema is a contract.** `company_rollup` mirrors a
> real external export byte-for-byte. When geography was needed for the
> territory map, it was **derived** from the account name
> (`location_for_account`) rather than added as a column, precisely to keep that
> contract intact.

## 7. The account-intelligence pipeline (вҖңstrategizeвҖқ)

"Sort accounts into cadences" is the heart of the app. It runs in two worker
functions, driven from the Accounts tab, narrated live via `_GMA_STATE` /
`_STRATEGIZE_STATE` (which the UI polls). Output is written to
`Account_Intelligence/output/*.json` and the sub-apps' `output/latest.xlsx`.

### Phase A вҖ” import (`_run_get_my_accounts`)

`resolve вҶ’ isc вҶ’ ibm вҶ’ segment`

1. **Resolve** the seller's territory. `seller_accounts.resolve_seller(email)`
   maps the signed-in email to coverage IDs (CovIDs). Any email works; a
   non-rep gets a stable demo territory (`fake_data.demo_covids_for_email`).
2. **ISC** вҖ” pull the accounts in those territories (mock ISC/Sales Cloud).
3. **IBM** вҖ” join the IBM install base (for whitespace).
4. **Segment** вҖ” `Account_Segmentation` writes `output/latest.xlsx`
   (sheet `Segmented Accounts`) вҖ” the ~1,900-account book.

### Phase B вҖ” strategize (`_run_strategize`)

`contacts вҶ’ scoring вҶ’ quarters вҶ’ cadences вҶ’ distribute` вҖ” five stages, each
writing its own artifact so you can always see *why* an account landed where it
did:

1. **Contacts.** Every account is checked for a real IT decision-maker
   (`_has_decision_maker` via ZoomInfo). Those without one go to the **No
   contacts** list вҖ” you don't work an account you can't reach. вҶ’ `no_contacts.json`
2. **Scoring.** `Account_Tiering` (subprocess) scores the rest on IBM spend +
   trend, install base, company size, and recent buying signals. Writes
   `Account_Tiering/output/latest.xlsx` (sheet `Tiered Accounts`).
   > **Sizing gotcha.** Tiering has a demo throttle `RANDOM_SAMPLE_SIZE`
   > (env, default **50** in `Account_Tiering/config.py`, raised for the 1,911-account
   > book). At the old default only 50 accounts were ever scored, so only a
   > handful reached a cadence. If coverage looks tiny after enlarging the book,
   > this is why.
3. **Quarter segmentation.** Has-contacts accounts are split across the four
   quarters by buying signal (`_quarter_for`). Only the current quarter
   continues; the rest вҶ’ `other_quarters.json` (the **Future quarters** list).
4. **Cadence + rank + tag.** Each current-quarter account is matched to a
   cadence by its play (`_PLAY_TO_CADENCE`), ranked within it, and tagged
   (`_account_tags`). Accounts past the per-cadence cap
   (`_CADENCE_CAP = 60`) drop to **Leftovers** (`leftovers.json`) and rejoin the
   pool for a future quarter. Result вҶ’ `latest.json` (`{cadences: {name: [members]}}`).
5. **Distribute** (`_build_quarter_schedule`) вҖ” В§8.

The lists you see in the Accounts sidebar map 1:1 to these artifacts:
`latest.json` (the cadences), `leftovers.json`, `no_contacts.json`,
`other_quarters.json`.

## 8. The schedule builder

`_build_quarter_schedule(cadences, cur_q)` turns the strategy into a
day-by-day plan: `{ "days": { "YYYY-MM-DD": [activity, вҖҰ] } }`, where each
activity is `{account, cadence, type: "email"|"call", step}`.

How it lays out:

- **Starts at the top of the quarter**, not today. A seller is mid-quarter, so
  the weeks already behind them should show the work that *happened*, not a blank
  grid. Weekends stay empty (real behaviour).
- Cadence starts are **interleaved by rank** вҖ” every cadence's #1 account first,
  then every #2, вҖҰ вҖ” so each cadence's best accounts start earliest.
- `_STARTS_PER_DAY = 3` accounts begin their cadence per weekday; each account's
  email/call touches land on its cadence's own step days (`salesloft_cadence_steps`).
- `_next_weekday` skips Saturday/Sunday everywhere.

This one structure feeds the Schedule calendar, the Dashboard activity chart, the
Email/Call today-lists, and the meeting-stat derivation.

## 9. Derived data: geography & meetings

Two dashboard features have **no source system at all** and are derived
deterministically. Both are documented in-app (Settings) and here, because
mistaking them for pipeline data would be wrong.

### Geography (territory map)

The ISC export has no state/city column (В§6). Territory and city are **derived
from the account name** via `fake_data.state_for_account` / `city_for_account`,
the same rule the generator used, so the map agrees with the accounts list. The
`/api/territory` endpoint aggregates per-territory and per-city counts; the
Profile choropleth renders four hand-traced outline maps (CA/HI/GU/MP) with
discrete region shading and a hover tooltip.

> The territory outlines are **recognisable approximations, not survey-grade
> boundaries** вҖ” the app ships no geo dataset. Regions are real places in real
> order (latitude bands for CA/GU, actual islands for HI/MP), but boundaries are
> approximate. No county lines are invented.

### Meetings & opportunities

`_meeting_stats(scope, offset)` derives meeting tallies from the **outbound
schedule** вҖ” there is no meetings source in ISC or the mock Salesloft. The
model: an account that's been worked can book a meeting; a meeting lands ~a week
after first touch; it counts as *completed* only once that date has passed; a
fraction cancel; a fraction of held meetings produce an opportunity (OI) with a
value. **All rates are fixed and every draw is seeded by account name**, so the
tallies are stable across restarts rather than drifting each reload. Scope can be
the whole quarter or a single week (with prev/next stepping).

Treat these as **illustrative**, not pipeline data.

## 10. The HTTP API

54 routes. The ones that matter for understanding the app:

**Pages / status**
- `GET /` вҖ” the SPA. `GET /api/status`, `/api/state`, `/api/seller` вҖ” process/seller state.

**Import & strategize** (async; UI polls state)
- `POST /api/get_my_accounts/run`, `POST /api/strategize/run` вҖ” the two workers (В§7).

**Accounts**
- `GET /api/accounts/list` вҖ” the whole book with tags, cadence, bucket.
- `GET /api/accounts/detail?name=` вҖ” one account, every source system + AI analysis.
- `GET /api/accounts/details?names=` вҖ” **batched** detail (unit-separator-joined names), for today-lists (В§5).
- `GET /api/accounts/leftovers | /no_contacts | /other_quarters` вҖ” the set-aside lists.

**Schedule / dashboard**
- `GET /api/schedule` вҖ” the full `{days:{iso:[вҖҰ]}}` structure (В§8).
- `GET /api/dashboard` вҖ” today's counts + items.
- `GET /api/dashboard/progress?period=&offset=` вҖ” the Activity chart. `period вҲҲ {day,week,month,quarter}`; `offset` steps the window (вҲ’1 = previous). Buckets by the axis that carries information at that grain (quarterвҶ’months, weekвҶ’weekdays, dayвҶ’by cadence).
- `GET /api/book?mtg_scope=&mtg_offset=` вҖ” book-of-business + meeting stats (В§9).
- `GET /api/territory?view=` вҖ” per-territory + per-city aggregates (В§9). `view вҲҲ {accounts,cadences,spend,industries}`.

**Email / call**
- `GET /api/email_draft`, `GET /api/call_brief` вҖ” draft an email / a pre-call brief for one account.

**Cadences** вҖ” `GET /api/cadences`.

**Assistant** (В§12)
- `POST /api/assistant/message` вҖ” one chat turn. Always 200 (fail-soft).
- `GET /api/assistant/status` вҖ” whether the service is configured.

**Mocked sign-in / sub-app process control** вҖ” the `/api/login/*`,
`/api/credentials/*`, `/mock/*`, and `/api/step*/run` families drive the mocked
authentication and the sub-app subprocesses.

Every route passes through the **`@app.before_request` auth guard** (В§14).

## 11. AI integration 1 вҖ” watsonx.ai (Granite)

`llm_advisor.py` is the client for the **generative** model
(`ibm/granite-4-h-small` by default). Used for the account **Play + Sales Angle**
(`advise_accounts`), **pre-call briefs** (`advise_call_brief`), and **email
drafts** (`advise_email`, via `/api/email_draft` вҖ” fail-soft to
`_deterministic_email`).

Mechanics:

- **Auth** вҖ” `_get_iam_token` exchanges `WATSONX_API_KEY` for an IAM bearer
  token at `iam.cloud.ibm.com`, cached ~55 min, thread-safe (Bobby drafts from a
  thread pool).
- **Call** вҖ” `_complete` / `complete_with_meta` POST a chat completion to
  `WATSONX_URL` for `WATSONX_PROJECT_ID`, stdlib `urllib` only.
- **Fail-soft** вҖ” returns `None` on *any* problem (missing key, over quota,
  network, non-200, malformed body). Callers fall back to deterministic output
  built from the same account data, so the feature degrades instead of breaking.
- **Telemetry** вҖ” `last_call_info()` / `_record_call` expose whether a *real*
  request was made, so the UI's AI-activity claims are backed by an actual call,
  not a static label.

Config: `WATSONX_API_KEY`, `WATSONX_PROJECT_ID`, `WATSONX_URL`
(default `https://us-south.ml.cloud.ibm.com`), `WATSONX_MODEL_ID`.

## 12. AI integration 2 вҖ” watsonx Assistant

`assistant.py` is the client for the **conversational** service вҖ” the "Ask
BobBee" help panel. It talks to the **Assistant v2 REST API**.

**Scope is deliberately narrow and privacy-preserving: the seller's book is
never sent.** Only a short product-description string (`_SYSTEM_CONTEXT`) leaves
the machine вҖ” no account names, contacts, spend, or schedule. This keeps the
"runs with no internet / nothing leaves the machine" property intact.

Mechanics (mirrors `llm_advisor` deliberately):

- **Auth** вҖ” same IAM token-exchange pattern.
- **Sessions** вҖ” the v2 API is session-based. `_session_for(client_id)` caches
  one session per browser client (multi-turn context), rotating before the
  ~30-min server timeout. A dead session reads as HTTP 404 and is **rebuilt once**
  rather than surfaced as an error.
- **Fail-soft** вҖ” unconfigured or unreachable, `ask()` returns a **local
  fallback answer** for common topics (cadences, leftovers, no-contacts,
  scoring, AI usage) and sets `live: false`. The UI **labels those replies as
  offline** so a canned string is never passed off as the assistant's answer.

Config (all from the Assistant instance in IBM Cloud вҖ” see the README / this
doc's intro for where each is found):

| Var | What it is |
|-----|-----------|
| `WXA_API_KEY` | IBM Cloud API key with access to the Assistant service. |
| `WXA_SERVICE_URL` | `https://api.<region>.assistant.watson.cloud.ibm.com/instances/<id>` |
| `WXA_ASSISTANT_ID` | The v2 assistant / environment ID. |
| `WXA_API_VERSION` | Optional; defaults to `2024-08-25`. Verify against your instance. |

Routes: `POST /api/assistant/message`, `GET /api/assistant/status`.

> **The two AI integrations are independent.** watsonx.ai (Granite) does
> generation; watsonx Assistant does conversation. Either can be configured
> without the other, and both fail soft.

## 13. The design system & theming

All styling is CSS custom properties in the `:root` block of `ui_templates.py`,
following **IBM Carbon**.

- **Palette** вҖ” Carbon **Gray 100** dark theme (neutral, *not* cool gray вҖ” the
  latter reads green-tinted). Brand **Blue 60** (`#0f62fe`) and **Purple 60**
  (`#8a3ffc`), with the blueвҶ’purple gradient as the brand accent.
- **The colour convention (В§2)** is encoded here: `--sig-*` (account signals) and
  `--viz-*` (chart stops) stay on the blue/purple/cool-gray ramp; **intensity**
  carries meaning (light = opportunity, deep = risk) rather than stoplight
  red/green.
- **Type** вҖ” IBM Plex Sans / Mono. Sharp, rectangular corners (`--r-*: 0`).
- **Light mode** вҖ” a full `:root[data-theme="light"]` override (Carbon
  White/Gray 10 layering). Toggled in **Settings вҶ’ Appearance**, persisted to
  `localStorage`, and **applied by an inline `<head>` script before first paint**
  so light-mode users never see a dark flash. The header stays dark in both
  themes (Carbon UI Shell convention), so its text is pinned rather than
  following `--text1`.

Because everything routes through tokens, a new theme is a new `:root[...]`
block вҖ” no component-level changes.

## 14. Security model

The full authorization audit is in [SECURITY.md](SECURITY.md); the essentials:

- **Loopback only.** Every server binds `127.0.0.1`. Not LAN-exposed.
- **Loopback is not a trust boundary**, though вҖ” any web page the seller visits
  can `fetch()` `127.0.0.1`, and a DNS-rebinding page can point its hostname
  there. The **`@app.before_request` guard** (`_dashboard_auth_guard`) closes
  both:
  - **Host-header check** вҖ” `Host` must be loopback, or 403 (anti DNS-rebinding).
  - **CSRF** вҖ” state-changing requests (`POST/PUT/DELETE/PATCH`) with a
    non-loopback `Origin`/`Referer` are blocked. Same-origin UI POSTs carry a
    loopback origin and just work.
  - Optional **strict mode** (`DASHBOARD_STRICT_AUTH=1`) additionally blocks
    tokenless, origin-less mutating requests (a local curl/script).
- **Mocked sign-in.** One IBM W3ID login (mocked) covers ISC/ZoomInfo/Salesloft.
  Only the email is kept locally (`credential_store`); the password is
  discarded. The Assistant sends none of this.
- **File hardening** вҖ” session/secret files are re-secured to owner-only on
  launch (`guard.harden_perms`).
- **No interactive debugger** вҖ” `debug=False` even with the reloader on.

## 16. Testing

`npm test` вҶ’ `pytest tests/ -q`. Three suites:

- **`test_dashboard.py`** вҖ” the deterministic fake-data generators and joins.
  Notably enforces the **ISC rollup schema is byte-faithful** (23 columns, the
  trailing-space header). This is the test that stops geography from being
  bolted onto the export as a column (В§6).
- **`test_run_pipeline_auth.py`** вҖ” the Host-header / CSRF guard (В§14):
  DNS-rebinding hosts get 403, cross-origin mutations blocked, same-origin
  allowed.
- **`test_guard.py`** вҖ” file-permission hardening.

All deterministic вҖ” no network, no model calls. A model integration failing
must never fail the suite, because the app is defined to work without it.
